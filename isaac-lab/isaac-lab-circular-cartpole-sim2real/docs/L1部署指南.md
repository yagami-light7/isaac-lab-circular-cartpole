# L1 倒立摆部署指南

本文档基于当前仓库已有代码，说明如何把 `isaac-lab-circular-cartpole-sim2real` 部署成一条可落地的 `L1` 实机控制链路。

目标链路固定为：

- 仿真与训练：Isaac Lab
- 策略导出：`policy.onnx` 或 `policy.pt`
- 主机推理：`host_runtime`
- 主机与板端通信：USB CDC
- 下位机：MC02
- 电机总线：FDCAN
- 电机：DM4310
- 被动关节编码器：MT6701

## 1. 当前代码已经包含什么

当前仓库里和 `L1` 实机部署直接相关的文件如下：

- 环境注册：`source/circular_cartpole_sim2real/circular_cartpole_sim2real/tasks/manager_based/circular_cartpole_sim2real/__init__.py`
- `L1 real` 环境：`source/circular_cartpole_sim2real/circular_cartpole_sim2real/tasks/manager_based/circular_cartpole_sim2real/s2r_task_l1_real_v0.py`
- `L1 real` 的 RSL-RL 配置：`source/circular_cartpole_sim2real/circular_cartpole_sim2real/tasks/manager_based/circular_cartpole_sim2real/agents/rsl_rl_ppo_cfg_l1_real_v0.py`
- 训练脚本：`scripts/rsl_rl/train.py`
- 回放与导出脚本：`scripts/rsl_rl/play.py`
- 主机运行时：`host_runtime/`
- 二进制协议定义：`host_runtime/protocol.py`
- 主机 history 拼接：`host_runtime/history.py`
- 主机推理与串口联调：`host_runtime/cli.py`
- 部署契约摘要：`docs/L1_REAL_DEPLOYMENT.md`

当前已经约定好的 `Gym` 任务 ID 是：

```text
RK-Circular-Cartpole-S2R-L1-REAL-V0
```

## 2. 先说结论：应该用哪套权重

你现在有三类权重来源，部署顺序建议如下：

1. 首选 up 主的 `sim2real L1` 权重。
2. 其次是你在当前 `L1 real` 环境上重新训练或微调出来的权重。
3. 不建议直接把 `isaac-lab-circular-cartpole` 里旧的 `L1/L2/L3` 权重直接上这条链路。

原因很简单：

- 当前实机主线已经固定成 `L1 real` 契约。
- 这个契约要求主机侧输入是 `5` 个特征乘以 `3` 帧 history，共 `15` 维。
- 动作语义是“每个控制周期的相对位置增量”。
- `rsl_rl_ppo_cfg_l1_real_v0.py` 已明确关闭了 actor/critic 观测归一化，所以主机侧不需要额外 normalizer。

如果 up 主的硬件、机构、传感器和你基本一致，那么最务实的顺序不是一上来就重训，而是：

1. 先用 up 主的 `sim2real L1` 权重把链路跑通。
2. 若能稳定站立，先完成系统联调。
3. 只有在“能跑但不稳”或者“你的观测/动作契约后续继续修改”的情况下，再回 Ubuntu 重训或微调。

## 3. 系统架构和职责划分

部署时分成三层：

### 3.1 Ubuntu 主机

职责：

- 加载 `policy.onnx` 或 `policy.pt`
- 接收 MC02 上传的 `SensorFrameV1`
- 维护 `3` 帧历史
- 组装 `15` 维观测
- 输出 `ActionFrameV1`
- 记录 JSONL 日志，供 replay 校验

### 3.2 MC02 下位机

职责：

- 通过 SPI 读取 MT6701
- 通过 FDCAN 读取并控制 DM4310
- 统一零位、符号、角度 wrap 规则
- 本地估算角速度
- 在 250 Hz 节拍下打包 `SensorFrameV1`
- 接收 `ActionFrameV1`
- 执行 watchdog、急停、超温、异常保护

### 3.3 DM4310

建议首版使用 MIT 位置跟踪形式，而不是原始力矩模式：

- `p_des = 当前参考位置 + base_pos_delta_rad`
- `v_des = 0`
- `kp, kd` 由上位机下发或由板端配置
- `t_ff = 0`

也就是说，当前主机运行时输出的是“相对位置增量”，不是绝对目标角，也不是直接力矩。

## 4. 观测与动作契约

这一节是整个项目最关键的部分。权重能不能直接复用，首先看这里是否一致。

### 4.1 单帧观测顺序

主机每次从 `SensorFrameV1` 提取 5 个特征，顺序固定为：

1. `base_pos_rad`
2. `flex1_pos_rad`
3. `base_vel_rad_s`
4. `flex1_vel_rad_s`
5. `last_action_delta_rad`

实现位置在：

- `host_runtime/history.py`

### 4.2 History 展开顺序

当前代码不是按“逐帧拼接”给网络，而是按“逐特征拼接”。

假设历史长度为 `3`，则最终 `15` 维观测为：

1. `base_pos[t-2], base_pos[t-1], base_pos[t]`
2. `flex1_pos[t-2], flex1_pos[t-1], flex1_pos[t]`
3. `base_vel[t-2], base_vel[t-1], base_vel[t]`
4. `flex1_vel[t-2], flex1_vel[t-1], flex1_vel[t]`
5. `last_action[t-2], last_action[t-1], last_action[t]`

如果 history 未填满，则每一组特征单独左侧补零。

这意味着：

- 板端不能直接把 3 帧数据拼给上位机
- 板端只上传“当前真实测量值”
- history 必须由主机维护

### 4.3 动作语义

当前 `L1 real` 环境里，动作被定义成：

- `base_to_fixed` 关节的相对位置命令

也就是说，策略网络输出的第一个值应解释为：

```text
base_pos_delta_rad
```

然后由板端把它转成电机位置参考的增量。

## 5. 二进制协议

协议定义在：

- `host_runtime/protocol.py`

所有帧均为小端序，CRC 使用 `CRC-16/CCITT-FALSE`。

### 5.1 SensorFrameV1

长度固定为 `46` 字节，字段如下：

- `magic`
- `version`
- `kind`
- `frame_length`
- `seq`
- `t_us`
- `base_pos_rad`
- `base_vel_rad_s`
- `flex1_pos_rad`
- `flex1_vel_rad_s`
- `motor_temp_c`
- `status_flags`
- `crc16`

### 5.2 ActionFrameV1

长度固定为 `42` 字节，字段如下：

- `magic`
- `version`
- `kind`
- `frame_length`
- `seq`
- `t_us`
- `base_pos_delta_rad`
- `kp`
- `kd`
- `watchdog_ms`
- `mode_flags`
- `crc16`

### 5.3 必须统一的量纲

板端发给主机前，所有数据必须先统一成以下单位：

- 角度：`rad`
- 角速度：`rad/s`
- 时间戳：`us`
- 温度：`degC`

不要把原始编码器计数、圈数、RPM 或角度制直接送入主机运行时。

## 6. Ubuntu 侧部署

这一部分负责训练、回放和导出策略。

### 6.1 前提条件

你需要一套已经能正常运行的 Isaac Lab 2.3.0 / Isaac Sim 5.1.0 环境。

推荐做法是在 Isaac Lab 根目录下，把本项目以 editable 方式安装进去：

```bash
cd <IsaacLab-root>
./isaaclab.sh -p -m pip install -e /path/to/isaac-lab-circular-cartpole-sim2real/source/circular_cartpole_sim2real
```

如果你已经能正常运行本仓库原始任务，这一步可以跳过。

### 6.2 训练 `L1 real`

在 Isaac Lab 根目录执行：

```bash
./isaaclab.sh -p /path/to/isaac-lab-circular-cartpole-sim2real/scripts/rsl_rl/train.py \
  --task RK-Circular-Cartpole-S2R-L1-REAL-V0 \
  --headless
```

日志默认会落到：

```text
logs/rsl_rl/cartpole_s2r_l1_real_v0/<时间戳>/
```

如果你不打算现在重训，而是先试 up 主现成权重，这一步可以先跳过。

### 6.3 导出 `policy.onnx` 和 `policy.pt`

`scripts/rsl_rl/play.py` 在加载 checkpoint 后会自动导出 `policy.pt` 和 `policy.onnx`。

推荐直接指定 checkpoint 路径执行：

```bash
./isaaclab.sh -p /path/to/isaac-lab-circular-cartpole-sim2real/scripts/rsl_rl/play.py \
  --task RK-Circular-Cartpole-S2R-L1-REAL-V0 \
  --checkpoint /absolute/path/to/checkpoint.pt \
  --headless
```

导出结果会位于该 checkpoint 所在运行目录下的：

```text
exported/policy.onnx
exported/policy.pt
```

对于首版实机部署，优先建议用 `policy.onnx`，因为主机侧依赖更轻。

## 7. 主机运行时部署

这一部分负责接板子、做推理、发动作。

### 7.1 运行位置

`host_runtime` 是纯 Python 主机侧工具，不依赖 Isaac Sim 图形界面。

可以运行在：

- Ubuntu 实机控制主机
- Windows 联调主机

如果目标是最终实机闭环，建议放在 Ubuntu 主机上，减少后续环境切换。

### 7.2 依赖

如果用 `policy.onnx`，至少需要：

```bash
pip install numpy onnxruntime pyserial
```

如果你要加载 `policy.pt`，还需要安装 `torch`。

### 7.3 离线自测

在仓库根目录下，可以先不接板子做离线联调。

#### 正弦命令冒烟

```powershell
python -m host_runtime sine --steps 2000 --amplitude 0.05 --frequency-hz 0.5 --log logs\sine.jsonl
```

这一步用于验证：

- `host_runtime` CLI 能正常启动
- 协议对象、history、日志逻辑能正常工作

#### 策略推理冒烟

```powershell
python -m host_runtime policy --policy path\to\policy.onnx --steps 1000 --log logs\policy.jsonl
```

这一步用于验证：

- ONNX 能成功加载
- 主机能生成动作
- 记录的 observation 和 action 可回放

#### 回放校验

```powershell
python -m host_runtime replay --log logs\policy.jsonl --no-policy-check
```

如果要校验模型输出一致性，再加上 `--policy path\to\policy.onnx`。

## 8. MC02 固件需要满足什么

虽然下位机代码由你自己写，但如果想和当前主机代码直接配合，固件必须满足下面这些要求。

### 8.1 250 Hz 主循环

控制周期固定为：

```text
dt = 0.004 s
频率 = 250 Hz
```

每个控制周期执行：

1. 读取 DM4310 当前 base 角度和角速度
2. 读取 MT6701 当前 flex1 角度
3. 在板端估算 flex1 角速度
4. 组装 `SensorFrameV1`
5. 通过 USB CDC 发送给主机
6. 接收 `ActionFrameV1`
7. 将 `base_pos_delta_rad` 转成位置参考增量
8. 下发电机控制命令

### 8.2 角度零位和方向

这一点非常关键。即使硬件一致，只要零位或方向不一致，现成权重也会马上失效。

你需要保证：

- `base_pos_rad` 的正方向与仿真中的 `base_to_fixed` 一致
- `flex1_pos_rad` 的正方向与仿真中的 `fixed_to_flex_1` 一致
- upright 姿态对应的角度参考与训练时一致

不要直接把传感器原始零点拿来用。必须先通过静态摆位、手动拨动和日志对比，把零位、方向、跨 `pi/-pi` 的 wrap 规则统一好。

### 8.3 保护逻辑

板端至少应实现：

- USB 超时 watchdog
- 电机超温保护
- 编码器异常值保护
- FDCAN 掉线保护
- 急停

最简单的策略是：

- 超时或故障时，立即停止更新位置参考
- 视故障等级决定保持当前参考或直接失能电机

## 9. 实机联调顺序

不要一上来就接 RL 闭环。按下面顺序来最快。

### 第 1 步：USB CDC 打通

目标：

- 主机能持续收到完整 `SensorFrameV1`
- `seq` 连续递增
- 无 CRC 错误

你可以先不接电机，只发固定测试帧。

### 第 2 步：DM4310 闭环打通

目标：

- 能读回 base 角度和角速度
- 能下发 MIT 位置跟踪命令
- 固定小角度位置跟踪稳定

这一步先不要接策略。

### 第 3 步：MT6701 打通

目标：

- 能稳定读取单圈角度
- 手动拨动时角度方向正确
- 差分速度符号正确
- wrap 处理正确

### 第 4 步：主机正弦联调

用主机的 `sine` 模式：

```powershell
python -m host_runtime sine --port COM5 --baudrate 921600 --amplitude 0.03 --frequency-hz 0.5 --kp 1.0 --kd 0.0 --log logs\sine_hw.jsonl
```

在 Ubuntu 上则把 `COM5` 换成 `/dev/ttyACM0` 或实际设备名。

目标：

- 底座关节能稳定跟踪小幅正弦
- 无明显发散或高频抖动
- 板端与主机长时间运行不丢帧

### 第 5 步：策略闭环

确认正弦联调通过后，再运行策略模式：

```powershell
python -m host_runtime policy --policy path\to\policy.onnx --port COM5 --baudrate 921600 --kp 1.0 --kd 0.0 --watchdog-ms 100 --mode-flags 1 --action-limit-rad 0.08 --log logs\policy_hw.jsonl
```

第一轮务必加防护：

- 加 tether
- 先小 `action_limit_rad`
- 人手随时准备急停

### 第 6 步：回放校验

闭环后立刻检查日志：

```powershell
python -m host_runtime replay --log logs\policy_hw.jsonl --policy path\to\policy.onnx
```

这一步能帮助你判断问题出在：

- 串口乱包
- history 拼接错误
- 板端符号或零位错误
- 策略输出与实际发送动作不一致

## 10. 成功标准

建议把验收拆成三层：

### 10.1 协议层成功

- 250 Hz 下稳定通信 10 分钟
- 无 CRC 错误
- `seq` 无跳变

### 10.2 执行层成功

- 正弦位置参考稳定
- 小步阶无明显发散
- watchdog、急停、过温保护均有效

### 10.3 RL 层成功

- tethered `L1` 可稳定摆起或稳定保持
- 自由运行时可连续稳定工作
- 多次启动成功率可接受

## 11. 什么时候必须回 Ubuntu 重训

下列情况出现任意一条，就建议回 Ubuntu 重新训练或至少微调：

- 你修改了观测顺序
- 你修改了 history 拼接方式
- 你修改了动作语义
- 你的板端延迟明显大于 up 主原方案
- 你的零位定义与原模型差异较大
- 能站起来但鲁棒性明显不够

如果只是硬件几乎一致、协议一致、观测一致、动作一致，那么优先直接试现成 `sim2real` 权重，通常比你现在立刻重训更省时间。

## 12. 当前最推荐的执行顺序

如果你的目标是“尽可能快把 `L1` 跑起来”，我建议按下面顺序执行：

1. 用 up 主的 `sim2real L1` 权重导出 `policy.onnx`
2. 用 `host_runtime policy --steps ...` 做离线冒烟
3. MC02 先只做 `SensorFrameV1/ActionFrameV1`
4. 用 `host_runtime sine` 打通正弦跟踪
5. 上 tethered `L1`
6. 若稳定则先完成整个 L1 系统
7. 若不稳，再回 Ubuntu 在 `RK-Circular-Cartpole-S2R-L1-REAL-V0` 上训练你自己的权重

这条路线最快，也最符合当前仓库的代码状态。
